# Живой документ ДЗ: структурные ответы, ревью-цикл, чат — дизайн

> Дата: 2026-07-02. Статус: **design (утверждается)**. Тип: продуктовая фича Folio, расширяет M8 (кабинет ученика #16), закрывает часть #17. Реализацию НЕ начинать до аппрува спеки.

## 1. Обзор

Назначенное ДЗ перестаёт быть статичным PDF и становится **живым онлайн-документом на пару «ученик ↔ репетитор»**:

- Ученик вписывает ответы **по каждому вопросу** прямо в кабинете (по ссылке-токену, без логина).
- Репетитор смотрит ответы и оставляет **комментарий к каждому ответу** (без оценок ✓/✗).
- Цикл **итеративный**: ученик сдал → репетитор прокомментировал и вернул → ученик исправил → сдал снова → … пока репетитор не **примет** задание (терминальный статус).
- Плюс **чат/комментарии по всему заданию** — отдельный тред на это назначение, пишут оба.

Кому: репетитор (веб, под сессией) и его ученик (веб, по cabinet-токену).

## 2. На чём строим (существующее)

- `folio_homework_assignments` (миграция `20260615151554`): `status ∈ (assigned, submitted, reviewed)` default `assigned`; поля `due_date`, `tutor_comment`, `assigned_at`, `assigned_by`; M8 (`20260701120000`) добавил на ученика `cabinet_token` и на назначение `submitted_at`.
- Кабинет ученика (`/[locale]/s/[token]`): рендер текста задания, «Я сделал» (`assigned→submitted`), PDF, один `tutor_comment`. Скоуп — service-role по токену (`lib/cabinet/*`).
- Ревью репетитора: `reviewAssignment` (`→reviewed` + `tutor_comment`) в `lib/homework/assignments.ts`.
- `derive.ts`: `current = assigned|submitted`, `completed = reviewed`.
- Общий движок генерации `_shared/generate.ts` (по HTTP через `folio-generate`), учёт LLM-расхода `eb_llm_usage` (#23 Ф1, для бота; Folio-учёт — Ф2 #23).

## 3. Ядро: откуда берётся структура (itemization)

Сгенерённое ДЗ — **свободный текст**, разный по типам модулей (reading / vocabulary / переводы / глаголы). Разбор регулярками хрупок. Решение:

> **При назначении ученику — один LLM-проход «итемизация»** через общий движок: текст задания → структурированный JSON `items[]`. Каждый item: `task_label` (напр. «Task 1 · True/False»), `question_text`, `item_type` (`tf` | `mcq` | `open` | `gap` | `other`). Надёжно на всех типах модулей (структурирует LLM, не regex). Через `output_config.format` (structured outputs) на движке.

- **Где:** новый action `itemize` в `folio-generate` (рядом с `generate`/`edit`), вызывается из серверного экшена назначения. Или отдельная функция движка `itemizeHomework(content)`.
- **Стоимость:** ~$0.01–0.02 за назначение (учтётся, когда доедет Folio-учёт usage, #23 Ф2; `action='itemize'`).
- **Правки:** структура сохраняется в БД; при кривой разбивке репетитор сможет поправить/перегенерить (Ф позже, не в MVP).

## 4. Стейт-машина (итеративный цикл)

Расширяем `status`. Добавляем `returned` и `accepted`; существующий `reviewed` мигрируем в `accepted` (он был терминальным).

```
assigned  — назначено; ученик может отвечать; не сдано
   │ (ученик «Сдать»)
   ▼
submitted — сдано, ждёт репетитора; ответы заблокированы для ученика
   │                         ▲
   │ (репетитор «Вернуть»)   │ (ученик «Сдать» снова)
   ▼                         │
returned  — репетитор оставил комментарии, вернул; ученик снова редактирует ──┘
   │ (репетитор «Принять»)
   ▼
accepted  — финальный аппрув; терминально, документ read-only
```

- **Редактирование ответов учеником:** разрешено в `assigned` и `returned`; заблокировано в `submitted` (ждёт ревью) и `accepted` (терминал).
- **Комментарии репетитора:** в любой момент до `accepted`. Действия репетитора: «Вернуть на доработку» (`submitted→returned`) и «Принять» (`submitted→accepted`).
- **Чат:** пишут оба в любой момент (в т.ч. после `accepted` — обсуждение остаётся).
- **Миграция:** widen CHECK до `(assigned, submitted, returned, accepted)` (+ временно `reviewed` для обратной совместимости), затем `update ... set status='accepted' where status='reviewed'`. Widening набора значений безопасно для существующих строк.
- **derive/кабинет:** `current = assigned|submitted|returned`; `completed = accepted`.

## 5. Модель данных

Новые таблицы (все с RLS: воркспейс-изоляция через родителя-назначение; доступ ученика — через service-role в серверных экшенах, скоуп по cabinet-токену, как в M8).

**`folio_homework_items`** — вопрос + ответ ученика + комментарий репетитора (1:1 с вопросом):
```
id            uuid pk
assignment_id uuid → folio_homework_assignments(id) on delete cascade
idx           int              -- порядок
task_label    text             -- «Task 1 · True/False»
question_text text
item_type     text             -- tf | mcq | open | gap | other
student_answer text            -- пишет ученик
tutor_comment  text            -- 💬 пишет репетитор (per-item)
updated_at     timestamptz
```
> Ответ и комментарий — разные колонки → одновременная правка учеником и репетитором не затирает друг друга.

**`folio_homework_messages`** — чат по назначению:
```
id            uuid pk
assignment_id uuid → folio_homework_assignments(id) on delete cascade
author        text             -- 'student' | 'tutor'
body          text
created_at    timestamptz
```

RLS: обе таблицы `enable row level security`; политика воркспейс-изоляции через `assignment_id in (select id from folio_homework_assignments where workspace_id = folio_current_workspace_id())` (репетитор под сессией). Ученик читает/пишет через серверные экшены на service-role, резолвя `token→student→assignment` (без прямого клиентского доступа) — тот же приём, что в кабинете M8.

## 6. Поверхности (UI)

1. **Ученик (кабинет `/s/[token]`):** для активного ДЗ — рендер по `items` (task_label → вопрос → поле ответа + видимый `tutor_comment` под ним, если есть). Автосохранение ответа (debounce) или явная «Сохранить». Кнопка «Сдать» (когда статус позволяет). Блок чата снизу.
2. **Репетитор (в `AssignmentsList` / отдельная review-панель):** по каждому item — ответ ученика + поле комментария. Кнопки «Вернуть на доработку» и «Принять». Рядом — ключ из Teacher's Guide, если шаблон генерился с ответами (nice-to-have). Блок чата.
3. **Чат:** общий компонент, тред по `assignment_id`, обновление **по опросу/при заходе** (вебсокетов на Cloudflare Workers нет — поллинг каждые N сек, пока открыт).

## 7. Доступ и безопасность

- Ученик — **capability-токен** (`cabinet_token`): все его чтения/записи (ответы, чат от 'student', «Сдать») идут через серверные экшены на service-role, скоуп `token→student→assignment`. Прямого клиентского доступа к таблицам у ученика нет.
- Репетитор — сессия (anon+JWT) + RLS воркспейс-изоляция.
- Валидация на границах (zod): длины `student_answer`/`body`, `author` строго из контекста (не из тела), `item_id` принадлежит назначению.
- `#10` (rate-limit) актуализируется: чат/автосейв — новые часто-дёргаемые эндпоинты; при выходе на реальных людей — WAF-лимит (уже в беклоге).

## 8. Фазы (фича большая — режем на отдельные PR)

- **Ф1 — структура + ответы ученика:** itemization на назначении (движок + `itemize` в `folio-generate` + миграция `folio_homework_items`); кабинет рендерит items и сохраняет ответы; «Сдать». Стейт `assigned↔submitted`.
- **Ф2 — ревью репетитора + цикл:** per-item комментарии, «Вернуть»/«Принять», миграция статусов (`returned`,`accepted` + перенос `reviewed`), обновление derive/кабинета (read-only на `accepted`).
- **Ф3 — чат:** таблица `folio_homework_messages` + компонент треда (поллинг), обе стороны.
- **Ф4 — уведомления (опц.):** бот пингует репетитора о «Сдал», ученика о «Вернул/Принял» (через Telegram-связку). Позже.

## 9. Вне рамок / открытые вопросы

- Ручная правка кривой itemization репетитором — позже (Ф+).
- Автогенерация/показ ключа ответов ученику — нет (только репетитору, и то как nice-to-have Ф2).
- Реалтайм (вебсокеты) — нет; поллинг достаточно.
- Учёт стоимости `itemize` в `eb_llm_usage` — подключится с Folio-учётом (#23 Ф2).
- Групповые ДЗ: каждый ученик — своё назначение/свой документ (уже так по модели).

## 10. Глоссарий

- **Живой документ ДЗ** — назначение (`folio_homework_assignments`) + его `items` (вопросы/ответы/комментарии) + `messages` (чат).
- **Itemization** — LLM-разбор текста задания на структурированные вопросы.
- **Ревью-цикл** — `submitted ⇄ returned` до `accepted`.
- **cabinet_token** — capability-токен ученика для входа в кабинет без логина.
