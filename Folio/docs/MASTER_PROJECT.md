## 1. Что это

Folio — рабочая платформа репетитора. Не маркетплейс, не LMS в классическом смысле.
Операционная система для частного репетитора или небольшой школы:
ученики, расписание, учёт денег, домашние задания, прогресс — всё в одном месте.

**Первый клиент:** репетитор английского языка, уже использует [english-bot](https://github.com/GarroV/english-bot) для генерации домашних заданий. Folio расширяет этот инструмент до полноценной платформы.

**Целевая аудитория учеников:** подростки и молодёжь, RU/BY рынок.

## 2. Стек

> Актуальные версии и детали — в [ARCHITECTURE.md](ARCHITECTURE.md) и `Folio/package.json`.

| Слой | Технология |
|---|---|
| Фронтенд | Next.js 16 (App Router, src-dir) + React 19 |
| UI компоненты | shadcn/ui + Tailwind CSS v4 |
| Бэкенд | Supabase (Postgres + RLS + Edge Functions + Storage) |
| Язык | TypeScript (strict) |
| i18n | next-intl, локали `ru` (дефолт) / `en`, сегмент `[locale]` |
| Авторизация | Telegram (bot deep-link + одноразовый токен) — основной вход; email magic-link и Telegram Login Widget отложены |
| Генерация заданий (LLM) | Anthropic Claude (`claude-sonnet-4-6`) — общий движок с english-bot (`supabase/functions/_shared/generate.ts`), по HTTP через Edge Function `folio-generate` |
| Автоматизация (событийная) | n8n — **запланировано (V2+), пока не внедрено**; событийная логика сейчас в server actions / best-effort |
| Хостинг фронта | Cloudflare Workers через OpenNext (`@opennextjs/cloudflare`) |
| База данных | Тот же Supabase-проект, что у english-bot (`btlglelwxazdxfqdmcti`); изоляция через префикс таблиц `folio_*` |
| Часовой пояс | Московское время по умолчанию, поле `timezone` в профиле заложено (per-user TZ — отложено) |

> **Историческое примечание.** Исходный стек (ADR-001, 2026-06-04) предполагал хостинг на **Vercel** и **OpenAI GPT-4o-mini** для AI-фич. По ходу разработки решено: хостинг — **Cloudflare Workers/OpenNext** (см. ARCHITECTURE «Hosting»), генерация — на **общем движке Anthropic Claude** с ботом (решение о равносилии веб ↔ бот, M7a). OpenAI и n8n в коде пока отсутствуют.

## 5. Авторизация и инвайты

> Реализовано иначе, чем в исходном замысле: основной вход — **через Telegram** (bot deep-link + одноразовый токен), а не email. Детали флоу — в [ARCHITECTURE.md](ARCHITECTURE.md) и [ADR-003](003-telegram-auth.md).

- **Основной вход** — Telegram: страница `/login` минтит токен → бот ловит `/start folio_login_<token>` и подтверждает → сайт выпускает сессию Supabase. Работает без публичного домена на боте (не Login Widget).
- **Регистрация репетитора** — self-serve по инвайт-ссылке (`folio_signup_invites`): новый workspace + демо-данные, атомарно через RPC `folio_register_tutor`. Инвайты выпускает Super Admin в панели `/admin`.
- **Ученик** — создаётся репетитором в админке; собственный вход/инвайт ученика — **отложено** (M2/M8).
- **Email magic-link** и **Telegram Login Widget** — **отложено**.
- Таблица `folio_auth_methods` — не Telegram ID как PK, identity через внутренние `user_id`.

## 6. Модули

Архитектура — **модульный монолит**. Один деплой, код разбит по фичам.
Каждый модуль — своя папка, свои таблицы, свой слой API. Выносится в отдельный сервис без переписывания.

### 6.1 Students
- Профиль ученика: имя, контакты, Telegram ID, email, дефолтная ставка за урок
- Статус: активный / архивный
- При архивации: персональные данные удаляются, агрегированная статистика остаётся
- Привязка к workspace

### 6.2 Schedule (Расписание)
- Интерактивная доска в стиле Google Calendar
- Типы занятий: `solo` (1-1) и `group` (несколько учеников)
- Групповое занятие = одно событие + таблица `lesson_students` (ученик + сумма + статус)
- Действия: создать, перенести (изменить дату), отменить, отметить как состоявшееся
- При провале в занятие: все данные об ученике, история, прогресс, домашки
- Офлайн и онлайн занятия — оба учитываются, тип фиксируется

### 6.3 Billing Tracker (Учёт, не биллинг)
- Логика: `начислено по занятиям` − `зафиксированные оплаты` = `остаток`
- Дефолтная ставка хранится в профиле ученика
- В конкретном занятии ставку можно переопределить
- Смена ставки задним числом — только через редактирование конкретного занятия, не массово
- У репетитора: таблица по всем ученикам, фильтры по периоду
- У ученика: его баланс и история транзакций
- Оплата фиксируется репетитором вручную ("ученик передал X руб")

### 6.4 Homeworks (Домашние задания)
Две сущности:

**`homework_templates`** — контент и кэш генератора
- тема, уровень (A1/A2/B1/B2/C1/C2), тип (грамматика/лексика/чтение/письмо), сложность
- связь с english-bot: бот пишет сюда при сохранении, читает отсюда при кэш-хите

**`homework_assignments`** — назначение конкретному ученику
- `template_id` + `student_id` + `assigned_at` + `due_date` + `status`
- один шаблон можно назначить нескольким ученикам
- дедлайны и статус — управляет репетитор вручную

Доставка:
- Сейчас: PDF через Telegram бот или email
- В будущем: интерактивный просмотр в веб-кабинете ученика

### 6.5 Lesson Journal (Журнал урока)
- Заполняется репетитором после каждого занятия (prompt после отметки "состоялось")
- Поля: тема урока (текст), уровень (A1–C2), комментарий, опциональная оценка прогресса
- Источник для прогресса и ачивок
- История хранится навсегда (даже после архивации ученика — агрегат)

### 6.6 Progress + Achievements (Прогресс и ачивки)
Два источника данных:
1. Назначенные домашки (тема + уровень из template)
2. Записи в журнале урока (тема + уровень от репетитора)

Механика ачивок (V2, не реализовано):
- Триггер: новая запись в журнале или новая домашка
- LLM генерирует: название ачивки + emoji + короткое описание (движок — общий Anthropic Claude, как у генерации заданий)
- Примеры: "Мастер Past Perfect 🎓", "Знаток лондонского метро 🚇"
- Прогресс-бар: пройденные занятия / цель (устанавливает репетитор)
- Ачивки висят в профиле ученика, видны в его кабинете

### 6.7 Groups
- Групповое занятие — это `lesson` с несколькими `lesson_students`
- Репетитор собирает состав при создании группового занятия
- У ученика в расписании: пометка "групповое" vs "индивидуальное"
- Ученик не видит других участников группы

### 6.8 Notifications (Уведомления)
На старте: только база, без UI

- Таблица `notifications`: `user_id`, `channel` (telegram/email), `type`, `payload`, `sent_at`, `status`
- Абстракция канала: `NotificationService` с методами `send(channel, payload)`
- Реализации: TelegramChannel, EmailChannel
- Триггеры через n8n (см. раздел 9)

### 6.9 Template Editor (Редактор промптов)
- Репетитор редактирует промпты генерации домашек прямо в платформе
- english-bot читает промпты из Supabase, не из кода
- Версионирование промптов: `template_prompts` с историей изменений

### 6.10 Statistics (Статистика)
На старте: только модель данных, без UI

Агрегаты которые будем считать:
- Всего учеников (активных / архивных)
- Всего уроков (по месяцам)
- Общая сумма начислений / оплат
- Количество выданных домашек
- Средний уровень учеников по workspace
- Retention (сколько учеников активны >3 мес)

Принцип: **сырые данные хранятся в нормальных таблицах, агрегаты считаются через Postgres Views или materialized views**.

## 8. Bot Bridge (интеграция с english-bot)

> **Фактическое состояние (2026-06-30):** мост бот→веб работает — при скачивании PDF бот пишет задание в `folio_homework_templates` (`source='bot'`) воркспейса репетитора (резолв по верифицированной Telegram-связке). Генерация в вебе и в боте — на общем движке. **Авто-доставка ученику (n8n) и редактор промптов (`template_prompts`) — не реализованы** (M7c / V2). Канон — [ARCHITECTURE.md](ARCHITECTURE.md) «Homework generation». Ниже — исходный замысел прослойки.

english-bot остаётся как есть. Добавляем тонкую прослойку:

**При сохранении домашки в боте:**
- Бот пишет в `homework_templates` с `workspace_id` репетитора
- Поля: тема, уровень, тип задания, контент, source: 'bot'

**При назначении домашки ученику:**
- Создаётся запись в `homework_assignments`
- n8n триггер → отправка ученику (TG / email)

**Редактор промптов:**
- Бот читает промпты из `template_prompts` по `workspace_id`
- Репетитор меняет промпты в Folio UI

## 9. n8n — событийная логика

> **Статус: запланировано (V2+), не внедрено.** n8n в проекте пока нет. Синхронная логика живёт в Edge Functions и server actions; редкие побочные эффекты (начисление при «состоялось», мост бот→веб) сделаны best-effort прямо в действиях. Раздел ниже — целевой замысел событийной архитектуры.

n8n на Railway. Платформа дёргает n8n через webhook. n8n делает всё событийное.

| Событие | Что делает n8n |
|---|---|
| Урок отмечен "состоялся" | Начислить сумму → записать в журнал → триггер ачивки (OpenAI) → уведомление ученику |
| Домашка назначена | Отправить ученику (TG/email) → записать в историю |
| Еженедельно | Сводка репетитору: кто должен, кто не сдал домашку |
| Ученик создан | Отправить инвайт на email |
| После урока | Prompt репетитору заполнить журнал (через TG бота) |

**Принцип:** Edge Functions = API и бизнес-логика. n8n = всё событийное и асинхронное.

```
docs/

    └── ...
```

### Правила обновления
- **Каждая задача** включает обновление релевантных docs как часть Definition of Done
- **Каждое архитектурное решение** → новый файл в `decisions/`
- **BACKLOG.md** пополняется в любой момент, не теряем идеи
- **CLAUDE.md** содержит всегда-актуальные правила для Claude Code: миграции, i18n, запреты, паттерны

### CLAUDE.md — обязательные секции
- Стек и версии
- Правила миграций (никогда без локального теста)
- Правила безопасности (RLS обязателен, `workspace_id` проверяется на edge)
- i18n: ключи, файлы переводов, правило "новый текст = сразу RU+EN"
- Запреты: бизнес-логика не в боте/компоненте, всегда в shared слое
- Что делегируем Gemini, что нет

## 13. Ключевые принципы

1. **RLS с первого дня** — каждая таблица имеет политику по `workspace_id`
2. **Бизнес-логика не в компонентах** — логика в shared слое, UI — только отображение
3. **Документация как часть задачи** — не обновил docs = задача не done
4. **Модульность** — каждый модуль независим, граница чёткая
5. **Mobile-first** — вёрстка начинается с мобильного
6. **Component-driven UI** — shadcn/ui, тема меняется в одном месте
7. **n8n для событий** — асинхронное в n8n, синхронное в Edge Functions
8. **Investigate before touch** — перед рефакторингом reconnaissance в docs/
9. **Commits by risk** — новый код первым, production-критичное последним
10. **Pricing claims** — AI модели сверяются с официальными страницами провайдеров
