# ADR-002: Мультитенантность через workspace_id + RLS, общий Supabase-проект с english-bot

Date: 2026-06-04
Status: Accepted
Updated: 2026-06-08 — отказались от отдельного Supabase-проекта (см. ниже)

## Контекст

Folio должен поддерживать нескольких репетиторов в будущем. На старте — один.
Folio — не отдельный продукт, а расширение english-bot (тот же репозиторий, та же инфраструктура, тот же Supabase-проект `btlglelwxazdxfqdmcti`). Решение о выделении в отдельный Supabase-проект (исходная версия этого ADR) отменено: разработка продолжается в тех же ресурсах, где уже ведётся english-bot.

## Решение

- **Один Supabase-проект** (`btlglelwxazdxfqdmcti`) на оба продукта. Таблицы Folio — новые, с префиксом (например `folio_*`), не пересекаются с `eb_*`.
- Shared database, isolation между репетиторами через `workspace_id` на каждой таблице Folio + RLS политики Supabase. Воркспейсы создаёт Super Admin вручную.
- RLS-политики Folio пишутся независимо от политик english-bot — таблицы не пересекаются, конфликтов нет.

Не выбрали schema-per-tenant: избыточно для текущего масштаба, сложнее в обслуживании.
Не выбрали database-per-tenant: слишком дорого и сложно на старте.
Не выбрали отдельный Supabase-проект: лишняя инфраструктура и расходы при общем владельце и общих ресурсах.

## Последствия

- RLS обязателен на каждой таблице Folio — это правило в CLAUDE.md
- Миграции добавляют `workspace_id` с первого дня, даже если репетитор один
- Онбординг второго репетитора = создание записи в workspaces + инвайт, без изменений схемы
- Таблицы Folio именуются с префиксом `folio_` для явного разделения с `eb_*` (english-bot)
